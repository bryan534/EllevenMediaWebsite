The gap between the site's technical execution quality and its content depth is the defining SEO problem. ## Critical Issues — Fix Immediately

### CRITICAL-1: 3.5-Second `visibility: hidden` Preloader Kills LCP

**File:** `src/app.html` or layout component (line 48 of rendered homepage HTML)
**Impact:** Core Web Vitals — LCP guaranteed to fail on first visit
**Effort to fix:** 30 minutes

On every first visit to the site, this script fires before any content renders:

```javascript
try {
  if (!sessionStorage.getItem('elleven_preloader_seen')) {
    document.documentElement.style.visibility = 'hidden';
    setTimeout(function() {
      document.documentElement.style.visibility = ''
    }, 3500)
  }
} catch(e) {
  document.documentElement.style.visibility = ''
}
```

**What this does:** Sets the entire `<html>` element to `visibility: hidden` for exactly 3,500 milliseconds on every first pageview. Google's LCP timer begins at navigation start. This single script guarantees a Largest Contentful Paint of 3.5 seconds or more on all first visits — placing the site firmly in the "Needs Improvement" (2.5s–4s) or "Poor" (>4s) LCP band when accounting for any additional resource load time.

**Why this is the highest-priority technical fix on the site:** LCP is one of the three Core Web Vitals metrics Google uses as a direct ranking signal. A 3.5s floor before content is even visible to the browser's paint tracker means no optimization elsewhere can overcome this penalty. Fast TTFB, CDN delivery, and efficient CSS are all irrelevant if the document is hidden for 3.5 seconds.

**The problem with `visibility: hidden` vs `opacity: 0`:** Using `visibility: hidden` removes the element from accessibility and from the browser's LCP candidate tracking. Even if content technically loads during the 3.5 second window, the browser cannot register an LCP paint event until visibility is restored. `opacity: 0` behaves differently and may allow LCP detection in some browsers — but the real fix is to remove the preloader entirely or reduce it dramatically.

**Recommended fix:**
```javascript
// Option A: Remove entirely — cleanest solution
// (Delete the preloader script block from the layout)

// Option B: Reduce to ≤200ms if the animation is brand-critical
if (!sessionStorage.getItem('elleven_preloader_seen')) {
  document.documentElement.style.opacity = '0';
  setTimeout(() => {
    document.documentElement.style.transition = 'opacity 0.2s ease';
    document.documentElement.style.opacity = '1';
    sessionStorage.setItem('elleven_preloader_seen', '1');
  }, 50); // 50ms max, not 3500ms
}
```

**Expected LCP improvement:** -2.5s to -3.5s on first visit (from ~4s+ to sub-1.5s when combined with the LCP image preload fix below).

### HIGH-1: No HTML Edge Caching — Every Request Hits Origin

**Impact:** Global TTFB, performance
**Evidence:** `cf-cache-status: DYNAMIC` on all page responses
**Effort:** 30 minutes (Cloudflare dashboard or `wrangler.toml`)

Despite running on Cloudflare, the HTML document is not edge-cached. Every visitor globally makes a full round-trip to the origin server. The 107ms TTFB measured is from the San Jose edge — users in New York, London, or Sydney will see significantly higher TTFBs.

`cf-cache-status: DYNAMIC` means Cloudflare passes the request to origin without caching the response. This is the default behavior when no `Cache-Control` header is set on HTML responses.

**Fix — Option A: Add Cache-Control from SvelteKit origin:**
In `src/hooks.server.ts` (or equivalent):
```typescript
export const handle: Handle = async ({ event, resolve }) => {
  const response = await resolve(event);
  
  // Cache HTML pages at edge for 5 minutes, serve stale for 1 hour
  if (response.headers.get('content-type')?.includes('text/html')) {
    response.headers.set(
      'Cache-Control',
      's-maxage=300, stale-while-revalidate=3600'
    );
  }
  
  return response;
};
```

**Fix — Option B: Cloudflare Cache Rule (no code change):**
In Cloudflare dashboard → Rules → Cache Rules:
- Match: `hostname eq "ellevenmediagroup.com" and not starts_with(http.request.uri.path, "/cdn-cgi")`
- Then: Edge Cache TTL = 5 minutes, Browser Cache TTL = 1 minute

**Important:** Exclude `/contact` POST endpoint from caching to avoid serving cached form submission responses.

```typescript
          'vendor-svelte': ['svelte'],
        }
      }
    }
  }
});
```

2. Audit whether the motion library is needed on all pages or only on the homepage hero. If only the homepage, lazy-load it per-route.

3. Target: ≤5–6 initial JS module requests.

### HIGH-8: No LCP Image Preload

**Impact:** LCP (Largest Contentful Paint)
**Effort:** 30 minutes

No `<link rel="preload" as="image">` directive exists in the HTTP `Link` header or in the page `<head>` for the hero section's image or logo. The hero's primary visual element (the `logo-hero.svg` or the canvas-rendered plasma background) is not preloaded, meaning the browser discovers it only after parsing and partially executing the document.

For SvelteKit, the fix is to add a preload directive in the page's `<svelte:head>`:

```svelte
<!-- src/routes/+page.svelte -->
<svelte:head>
  <link
    rel="preload"
    as="image"
    href="/logo-hero.svg"
    fetchpriority="high"
  />
</svelte:head>
```

If the LCP element is the canvas (plasma background), add `fetchpriority="high"` to the canvas element and ensure any background image is preloaded similarly.

### MEDIUM-10: Cloudflare Rocket Loader Modifying Script Execution

Cloudflare Rocket Loader is active (`rocket-loader.min.js` is injected on all pages). Rocket Loader intercepts all scripts and changes their `type` attribute to a non-standard value (visible in the rendered HTML: `type="79808266ebe0c97b777caa69-text/javascript"`) before asynchronously re-executing them.

This can conflict with SvelteKit's own deferred module loading strategy, potentially causing hydration timing issues that affect INP. SvelteKit already handles JavaScript loading efficiently via native ES module preloading — Rocket Loader may be adding overhead rather than reducing it.

**Recommended action:** Disable Rocket Loader in Cloudflare Speed → Optimization → Rocket Loader and measure the before/after impact on INP and LCP via PageSpeed Insights. If performance improves (likely), keep it disabled.

### LOW-6: `<link rel="preload">` Uses Relative Paths

The CSS preload links in the HTTP `Link` header use relative paths:

These are `nopush` (correct — HTTP/2 server push is deprecated), but relative paths in preload hints can occasionally cause issues with certain CDN configurations. Absolute paths are safer:

This is a low-risk, low-priority item.
